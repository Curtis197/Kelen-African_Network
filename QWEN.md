## Qwen Added Memories
- Debug Protocol: Always use instrumentation-first approach. When debugging issues, NEVER change logic code until after adding strategic console.logs, running the app, collecting exact error data, and identifying the root cause with certainty. This saves 85% tokens and prevents guess-and-check cycles. Add diagnostic logs at every decision point, ask user to run and share console output, then analyze data before making any fixes.
- Debug logs are PERMANENT infrastructure. NEVER remove console.log statements after fixing an issue. Logs stay in code until final release (and maybe even after). Benefits: 1) Future bugs found instantly, 2) Production debugging, 3) Code documentation, 4) Regression detection. Adding logs is an investment, not a temporary tool.
- INSTRUMENT SKILL: When implementing/building/updating ANY code (components, functions, queries, forms, auth, etc.), ALWAYS add comprehensive console.logs on FIRST WRITE. Never write unlogged code. Log all inputs, outputs, errors, decisions. RLS gets special treatment (check error code 42501 explicitly). Logs are permanent infrastructure, never removed. This prevents time waste from blind debugging. Trigger: create, implement, build, add, update, modify.
- Supabase Storage RLS Policy Issue: When storage.objects policies have `WITH CHECK = null` on a `FOR ALL` policy, INSERT operations fail. Fix by explicitly adding `WITH CHECK` clause matching the `USING` clause. The `portfolios_admin_v2` policy needed this fix via migration `20260410000002`. Also, overly restrictive folder structure checks in INSERT policies can cause false negatives - use simple `bucket_id` check for upload policies when path structure is controlled by frontend (migration `20260410000003`).
- RLS LOGGING: Always log TWO types of RLS violations explicitly: 1) EXPLICIT RLS (error code 42501) - query blocked with error, 2) SILENT RLS (no error but 0 rows) - query succeeded but data filtered. Both types get detailed logging with table name, user, violation type, and fix instructions. Never assume 0 rows means empty table - always check for silent RLS filtering.
- DATABASE REFERENCE: Always check `.qwen/skills/instrument/DATABASE-REFERENCE.md` before writing any database code. Key tables: `user_projects` (client projects), `pro_projects` (professional portfolio), `professionals` (profiles), `project_logs` (daily logs). Never guess table names - verify from reference. Full schema in `supabase/database-scheme.sql`.
- PROJECT TABLES CLARIFIED: 1) user_projects = clients on platform managing their own projects, 2) pro_projects = pros managing client work (client may NOT be on platform, NOT displayed on portfolio, PRIVATE management tool), 3) professional_realizations = FINISHED work showcased on PUBLIC portfolio (what visitors see). Flow: pro_projects completed → professional_realizations created → displayed on portfolio.
- MIGRATION MAINTENANCE: After EVERY database migration, MUST update both: 1) supabase/database-scheme.sql, 2) .qwen/skills/instrument/DATABASE-REFERENCE.md. Never run migration without updating docs. All three must match: Actual DB = Scheme File = Reference File. See .qwen/skills/instrument/MIGRATION-MAINTENANCE.md for full protocol.
- RLS MAINTENANCE: After ANY migration that modifies RLS policies, MUST: 1) Export all policies via pg_policies query, 2) Update supabase/RLS-list.md with fresh export, 3) Review supabase/RLS-audit-findings.md for new issues. Keep RLS list always current. Query: SELECT tablename, policyname, cmd, roles, qual, with_check FROM pg_policies WHERE schemaname='public' ORDER BY tablename, policyname;
- SOURCE OF TRUTH: The actual database is ALWAYS the source of truth. Files are just documentation that must match DB: 1) supabase/database-scheme.sql = documentation reference, 2) .qwen/skills/instrument/DATABASE-REFERENCE.md = documentation reference, 3) supabase/RLS-list.md = documentation reference. If files disagree with actual DB, DB wins - update the files.
